# Com_Moon Agent 탭 MVP UI / 제품 스펙

## 1. 문서 목적

이 문서는 Com_Moon Hub 안에 `Agent` 탭을 신설하거나, 현재의 `/dashboard/ai` 콘솔을 `Agent` 탭으로 재정의하는 방향을 실제 제품 설계 수준으로 구체화한다.

정리 범위:

- 왜 Agent 탭이 필요한가
- 사용자가 이 탭에서 실제로 하려는 일은 무엇인가
- MVP에 반드시 들어갈 화면과 상호작용
- 캐릭터화 / 사무실화의 적정 강도
- 검증 지표와 후속 확장 우선순위

이 문서는 현재 존재하는 `apps/hub/app/dashboard/ai/*` 기반 화면과 Hub 정보구조를 전제로 작성한다.

## 2. 한 줄 제안

`Agent 탭`은 과거와 현재의 에이전트를 한 화면에서 보고, 적합한 에이전트를 빠르게 다시 호출하고, 이전 컨텍스트를 바탕으로 새 오더를 내리는 운영 레이어다.

핵심은 `AI를 보여주는 탭`이 아니라 `일하는 존재들을 지휘하는 탭`이다.

## 3. 왜 이 탭이 필요한가

현재 대부분의 에이전트 경험은 세션 중심이다.

- 방금 열어둔 대화는 기억나지만 예전에 잘 쓰던 에이전트는 다시 찾기 어렵다
- 같은 종류의 일을 다시 맡기고 싶어도 누구에게 시켜야 하는지 떠올리기 어렵다
- 여러 에이전트가 동시에 움직일 때, 상태와 역할을 한눈에 보기 어렵다

사용자 머릿속에서는 에이전트가 단순한 채팅 로그가 아니라 "이전에 같이 일한 작업 파트너"로 저장된다.

Agent 탭은 이 mental model을 제품이 받아주는 화면이어야 한다.

## 4. Agent 탭의 핵심 Job-to-be-Done

사용자는 이 탭에서 아래 4가지를 하고 싶다.

- 지난번에 잘 썼던 에이전트를 다시 찾기
- 지금 상황에 맞는 에이전트를 빠르게 고르기
- 한두 번의 조작으로 새 오더를 내리기
- 누가 바쁘고, 누가 멈췄고, 누가 바로 투입 가능한지 보기

즉 이 탭의 질문은 아래 4개다.

- 지금 누가 일하고 있지
- 누가 어떤 일을 잘하지
- 지난번 그 일을 맡았던 애는 누구였지
- 지금 이 일을 누구에게 시키면 되지

## 5. 제품 원칙

### 실용 우선

캐릭터화나 사무실화는 기억성과 애착을 높이는 수단이다.

MVP에서는 장식보다 아래 기능이 먼저 살아야 한다.

- 역할이 바로 읽혀야 한다
- 최근 작업이 바로 보여야 한다
- 재호출과 오더가 바로 가능해야 한다

### 관계 우선

중요한 것은 세션 목록이 아니라 `관계가 이어지는 에이전트`다.

따라서 각 에이전트에는 최소한 아래 문맥이 붙어야 한다.

- 이름 또는 별칭
- 역할
- 최근 작업
- 자주 맡는 일
- 마지막 활동 시점
- 현재 상태

### 지휘감 우선

이 탭은 감상용이 아니라 운영용이어야 한다.

화면은 항상 `상태 + 맥락 + 다음 액션`의 3점 세트로 구성한다.

### 게임화는 얕게

첫 단계에서 캐릭터를 너무 세게 밀면 업무 도구가 아니라 장난감처럼 보일 수 있다.

MVP에서는 아래 정도만 허용한다.

- 책상
- 배지
- 도구 아이콘
- 표정 또는 상태 조명

풀 캐릭터 일러스트나 과한 세계관 카피는 후순위다.

## 6. 정보구조 제안

### 권장 방향

기존 `/dashboard/ai`를 유지하되, 제품 라벨과 내비게이션 명칭을 `AI Console`에서 `Agent` 또는 `Agents`로 재정의한다.

이유:

- 이미 에이전트, 오더, 카운슬, 챗 구조가 일부 존재한다
- 새 탭을 0부터 만드는 것보다 전환 비용이 훨씬 작다
- 사용자는 "AI"보다 "Agent"를 더 운영 단위로 이해하기 쉽다

### 탑레벨 위치

- `/dashboard/agents`
  또는
- 내부 구현은 `/dashboard/ai` 유지, UI 라벨만 `Agents`로 노출

MVP에서는 alias 전략이 가장 현실적이다.

### 서브탭 제안

- `Overview`
- `Roster`
- `Orders`
- `Threads`
- `Council`

설명:

- `Overview`: 현재 활동, 투입 가능 에이전트, 주요 오더 요약
- `Roster`: 이전에 일한 에이전트들의 목록과 역할 중심 탐색
- `Orders`: 새 오더 생성, 진행 중 오더 추적, 재오더
- `Threads`: 에이전트별 컨텍스트와 대화 이력
- `Council`: 여러 에이전트 간 논의와 비교 검토

## 7. MVP 화면 구조

### A. Overview

첫 화면은 "지금 누구를 부를 수 있고, 무슨 일이 열려 있는가"에 답해야 한다.

필수 블록:

- 활성 에이전트 수
- 지금 바로 투입 가능한 에이전트
- 진행 중 오더
- 최근 재호출된 에이전트
- 최근 완료된 작업

첫 CTA:

- `적합한 에이전트 찾기`
- `지난번 에이전트 다시 부르기`
- `새 오더 만들기`

### B. Roster

이 화면이 Agent 탭의 핵심이다.

에이전트를 단순 카드 목록이 아니라 `기억 가능한 존재`로 보여줘야 한다.

카드 필수 필드:

- 에이전트 이름
- 역할 태그
- 대표 전문분야 1~2개
- 최근 작업 제목
- 마지막 활동 시점
- 현재 상태
- 즉시 액션 버튼

즉시 액션:

- `다시 부르기`
- `비슷한 일 시키기`
- `이전 컨텍스트 보기`

정렬 / 필터:

- 최근 사용순
- 성격 또는 역할별
- 프로젝트별
- 현재 사용 가능 여부

### C. Orders

오더 화면은 "어떻게 시킬지"를 단순화해야 한다.

MVP에서 중요한 것은 자유 입력보다 `쉬운 명령 템플릿`이다.

추천 구조:

- `무엇을 시킬지` 선택
- `누구에게 시킬지` 추천
- `이전 컨텍스트를 붙일지` 토글
- `바로 실행` 버튼

오더 템플릿 예시:

- 리서치 정리
- 초안 작성
- 코드 수정
- 버그 조사
- 문서 업데이트
- 비교 검토

### D. Threads

스레드 화면은 채팅 자체보다 `컨텍스트 재사용` 가치가 중요하다.

각 스레드에 필요한 정보:

- 어떤 에이전트와의 대화인지
- 어떤 프로젝트 문맥인지
- 마지막 산출물이 무엇인지
- 지금 이어서 할 수 있는 액션이 무엇인지

### E. Council

Council은 여러 에이전트의 합의와 대립을 보는 레이어다.

MVP에서는 깊은 토론 UI보다 아래만 있어도 충분하다.

- 주제
- 참여 에이전트
- 현재 상태
- 최근 핵심 메시지
- 결론 또는 다음 결정 필요 항목

## 8. 캐릭터화 / 사무실화 방향

### 왜 필요한가

시각적 구분은 단순 미감이 아니라 검색 비용을 낮춘다.

사용자는 텍스트보다 다음을 더 잘 기억한다.

- 어디에 있었는지
- 어떤 분위기였는지
- 어떤 상징을 달고 있었는지

즉 사무실화는 "검색 인터페이스"이기도 하다.

### 캐릭터 강도 3단계

#### Level 1. Badge Office

가장 추천하는 MVP 단계.

- 각 에이전트를 책상/패널/좌석 단위로 표현
- 역할별 도구 아이콘 부여
- 상태에 따라 조명, 배지, 표정만 다르게 표시

장점:

- 기억성이 좋아진다
- 업무 도구 톤을 유지할 수 있다
- 구현과 유지가 가볍다

#### Level 2. Soft Persona

- 작은 아바타 또는 마스코트 도입
- 말투, 선호 업무, 강점이 카드에 붙음

장점:

- 재호출 욕구가 올라간다
- 제품 차별점이 생긴다

위험:

- 과하면 유치해 보일 수 있다
- 역할보다 캐릭터가 앞서면 실용성이 약해진다

#### Level 3. Full Office World

- 방 구조
- 자리 배치
- 캐릭터 애니메이션
- 에이전트 간 이동과 상호작용

이건 매력은 크지만 MVP에선 금지에 가깝다.

이 단계는 기능 검증이 끝난 뒤, retention 증폭 장치로 검토한다.

## 9. MVP 권장안

첫 버전은 `Level 1. Badge Office`가 가장 적합하다.

즉:

- 캐릭터 일러스트는 최소화
- 좌석 / 작업대 / 툴킷 / 상태 배지 중심
- 카드는 예쁘지만 여전히 업무 카드처럼 읽힘

한 줄로 말하면 `귀여운 오피스`가 아니라 `기억 잘되는 운영 레이아웃`이어야 한다.

## 10. 사용자 플로우

### 플로우 1. 지난번 에이전트 다시 부르기

1. 사용자가 Agent 탭 진입
2. `최근 함께 일한 에이전트` 영역 확인
3. 이전 작업을 보고 적합한 에이전트 선택
4. `이어서 시키기` 클릭
5. 이전 컨텍스트 포함 여부를 확인하고 실행

### 플로우 2. 지금 일에 맞는 에이전트 찾기

1. 사용자가 "문서 정리" 같은 작업 의도를 선택
2. 시스템이 적합한 에이전트 2~3명을 추천
3. 사용자가 상태와 최근 작업을 보고 한 명 선택
4. 템플릿 오더를 보정해서 실행

### 플로우 3. 여러 에이전트 비교 검토 시키기

1. 사용자가 주제를 입력
2. `Council로 보내기` 선택
3. 참여 에이전트 선택
4. 검토 결과와 합의 상태 확인

## 12. 측정해야 할 검증 지표

이 아이디어는 미감이 아니라 사용성으로 검증해야 한다.

핵심 지표:

- 새 에이전트 생성 대비 기존 에이전트 재호출 비율
- 오더 생성까지 걸리는 시간
- 적절한 에이전트를 고르기까지의 클릭 수
- 에이전트 카드 클릭 후 실제 실행 전환율
- 최근 사용 에이전트 영역 클릭률

정성 지표:

- 사용자가 에이전트를 역할로 기억하는가
- 사용자가 에이전트를 위치/외형으로 기억하는가
- "누구에게 시켜야 할지 모르겠다"는 순간이 줄어드는가

## 13. 실험 우선순위

### 실험 1. 재호출 중심 MVP

가장 먼저 봐야 할 실험.

- 최근 에이전트 목록
- 역할 태그
- 다시 부르기 버튼

이것만으로 재사용률이 오르면 방향이 맞다.

### 실험 2. 추천 오더 템플릿

- 자유 입력 없이 템플릿 오더만 제공
- 작업 유형별 추천 에이전트 노출

이 실험은 "명령 피로"를 줄이는지 본다.

### 실험 3. 사무실형 레이아웃

- 텍스트 리스트형 roster
- 좌석 기반 roster

둘의 선택 속도와 재호출률을 비교한다.

## 14. 구현 우선순위

### Phase A. 리네이밍 + 구조 정리

- `/dashboard/ai`를 `Agent` 라벨로 재정의
- 내비게이션과 헤더 카피 통일
- Overview / Orders / Threads / Council 구조 정리

### Phase B. Roster 강화

- 에이전트 카드에 최근 작업과 재호출 액션 추가
- 최근 사용 / 역할별 / 상태별 필터 추가
- 재사용 가능한 컨텍스트 요약 표시

### Phase C. 쉬운 오더 UX

- 오더 템플릿
- 추천 에이전트
- 이전 컨텍스트 첨부 토글

### Phase D. Light Office UI

- 좌석 또는 작업대 기반 시각화
- 상태 조명과 역할 오브젝트 추가

## 15. 최종 판단

이 아이디어는 충분히 밀어볼 가치가 있다.

하지만 승부처는 캐릭터화 자체가 아니다.

승부처는 아래 3가지다.

- 이전 에이전트를 쉽게 다시 찾을 수 있는가
- 누구에게 어떤 일을 시킬지 빨리 결정할 수 있는가
- 에이전트가 "세션"이 아니라 "지속적인 작업 파트너"처럼 느껴지는가

즉 Agent 탭은 예쁜 AI 탭이 아니라, `에이전트 운영체제의 시작점`이어야 한다.

## 16. 권장 다음 액션

바로 진행한다면 아래 순서가 가장 좋다.

1. 현재 `/dashboard/ai`를 기준으로 Agent 탭 리네이밍 시안 작성
2. `Overview + Roster + Orders` 3면만 먼저 와이어프레임 설계
3. Roster를 `리스트형`과 `라이트 오피스형` 2안으로 비교
4. 재호출 버튼과 쉬운 오더 템플릿을 MVP 핵심으로 고정

이 4개가 잡히면, 그 다음에야 캐릭터 강도를 올릴지 판단한다.
